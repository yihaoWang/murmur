  <name>Task 1: Fix hotkey modifier sync and remove dual hotkey conflict</name>
  <files>Typeness/App/TypenessApp.swift, Typeness/UI/HotkeyRecorderView.swift</files>
  <action>
In TypenessApp.swift `startHotkeyMonitor()`:
1. Replace the two manual keyCode assignments with a single call: `hotkeyMonitor.loadSettings(from: settingsStore)` — this applies BOTH keyCode AND modifiers from SettingsStore. Remove the two lines `hotkeyMonitor.toggleKeyCode = ...` and `hotkeyMonitor.pttKeyCode = ...`.

In HotkeyRecorderView.swift:
2. Remove the `KeyboardShortcuts.Name` extension block (lines 4-7) entirely — these register global shortcuts via the KeyboardShortcuts framework that conflict with the CGEventTap system. The recorder UI widget still works without Name declarations; it just becomes unregistered shortcuts used purely as a picker.
3. Change `HotkeyRecorderRow` to use `KeyboardShortcuts.Recorder("", onChange: { shortcut in syncToSettingsStore(shortcut: shortcut) })` instead of `KeyboardShortcuts.Recorder(for: name)`. Remove the `name` property. Pass the title via the `LabeledContent` wrapper as before. NOTE: If KeyboardShortcuts 1.15.0 API requires a `Name` for `.Recorder`, keep the Name declarations but add `KeyboardShortcuts.disable(.toggleMode)` and `KeyboardShortcuts.disable(.pushToTalk)` in TypenessApp.setupApp() to prevent the framework from intercepting keystrokes.

Why remove/disable: KeyboardShortcuts.Name with a default registers a global shortcut handler. Even without `onKeyDown`, the framework's internal NSEvent monitor can intercept keystrokes before CGEventTap sees them, causing duplicate or swallowed events.
